# Push-уведомления без карты и без Blaze — альтернативный вариант

Эта инструкция **заменяет** `STAGE6_PUSH_SETUP.md` (тот вариант через Cloud Functions требовал платный план Blaze и привязку карты — недоступно с картами РФ).

## Как это работает

Вместо постоянно работающего сервера Google (Cloud Functions) используется **GitHub Actions по расписанию** — он уже бесплатно собирает тебе APK, теперь будет ещё и каждые 5 минут проверять новые сообщения и рассылать push.

**Важное отличие от "настоящего" Cloud Functions варианта:** push приходят не мгновенно, а с задержкой **до ~5 минут** (иногда чуть больше — GitHub Actions не гарантирует точность расписания под нагрузкой). Для тестирования и небольшого использования — некритично.

Firestore и отправка push через Admin SDK (то, что использует скрипт) — **бесплатны на плане Spark**, карта не нужна вообще.

## Шаг 1. Секрет для сервисного аккаунта — тот же, что раньше

Если ты уже создавал `FIREBASE_SERVICE_ACCOUNT` Secret для Cloud Functions — он и здесь подойдёт, ничего заново делать не нужно.

Если ещё нет:
1. Firebase Console → ⚙️ **Project settings → Service accounts**
2. **Generate new private key** → скачается `.json`
3. Открой файл, скопируй весь JSON
4. GitHub-репозиторий → **Settings → Secrets and variables → Actions → New repository secret**
5. Name: `FIREBASE_SERVICE_ACCOUNT`, Secret: вставь JSON

## Шаг 2. Загрузить новые файлы в репозиторий

- `push-worker/index.js`
- `push-worker/package.json`
- `.github/workflows/send-push-notifications.yml`
- Обновлённый `app/src/main/java/app/yodo/messenger/data/repository/MessageRepositoryImpl.kt` (добавлено поле `notified`)

## Шаг 3. Включить workflow

GitHub Actions с `schedule` иногда не активируется автоматически при первой загрузке — на всякий случай:
1. Вкладка **Actions** → **Send Push Notifications**
2. Если видишь кнопку **"Enable workflow"** — нажми
3. Проверь вручную: **Run workflow** — посмотри в логах, что скрипт отработал без ошибок

После этого он будет запускаться сам каждые 5 минут, ничего делать не нужно.

## Шаг 4. Firestore — может понадобиться Collection Group индекс

Скрипт ищет сообщения **во всех чатах сразу** через `collectionGroup("messages").where("notified", "==", false)`. Firestore для такого запроса иногда требует отдельный индекс с областью "Collection group" (не "Collection", это разные типы индексов).

**Как узнать:** если в логах workflow (Actions → Send Push Notifications → открой запуск) увидишь ошибку вида `FAILED_PRECONDITION: The query requires an index` — там будет прямая ссылка. Перейди по ней, Firebase Console создаст нужный индекс за один клик, подожди пару минут (индексы строятся не мгновенно) и запусти workflow заново.

## Шаг 5. Пересобрать APK

Не забудь — клиентское приложение тоже изменилось (поле `notified` при отправке сообщения). Обнови файлы, пересобери APK как обычно.

## Как проверить, что всё работает

1. Отправь сообщение с одного аккаунта другому (оба должны быть авторизованы на разных устройствах/эмуляторах)
2. Подожди до 5 минут
3. Открой **Actions → Send Push Notifications** → последний запуск → в логах должно быть "Успешно отправлено: 1"
4. На втором устройстве должно появиться уведомление

## Если решится вопрос с картой в будущем

Файлы `functions/`, `firebase.json`, `.firebaserc` и `deploy-functions.yml` из прошлого этапа никуда не делись — если однажды появится способ активировать Blaze, можно переключиться на них для мгновенных push вместо задержки в 5 минут. Обе системы (Cloud Functions и push-worker) **не должны работать одновременно** — иначе push будут дублироваться. Пока используем только `push-worker`.
